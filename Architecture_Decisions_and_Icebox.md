> Обновлено: 01.03.2026

## Принятые архитектурные решения

### ADR-1: Монорепо (frontend + backend в одном репо)
**Решение:** Оставить всё в одной папке. `src/` — фронт, `backend/` — питон.  
**Почему:** Для одного разработчика монорепо быстрее в управлении. `npm run dev` запускает оба сервера через `concurrently`. Разделять смысла нет пока нет команды.  
**Альтернатива:** Отдельные репо — отклонено как излишняя сложность.

### ADR-2: React 19 вместо Vue 3
**Решение:** После перезапуска прототипа в Google AI Studio стек сменился с Vue 3 + Pinia на **React 19 + TypeScript**.  
**Почему:** Google AI Studio сгенерировал рабочий прото на React, который уже имел нужный функционал. Переписывать на Vue — потеря времени без выигрыша.  
**Следствие:** Все упоминания Vue/Pinia/Vue Router в старом ТЗ — устарели.

### ADR-3: Dual History Pattern (два разных «history»)
**Решение:** В проекте существуют две независимые системы истории:  
1. `CombatState.history: List[LogEntry]` — **нарративный лог** событий боя (что произошло)  
2. `history_stack: list` + `history_index: int` в `main.py` — **технические снэпшоты** для Undo/Redo (20 штук)  
**Почему:** Лог нужен мастеру для отчёта о бое и экспорта в Markdown. Undo/Redo — оперативный инструмент, не нужен в финальном стейте.  
**⚠️ Важно для разработки:** Cursor/AI не должен путать эти две системы. `history_stack` НЕ сериализуется в `CombatState`, он живёт только в памяти сервера.

### ADR-4: Asset Override Pattern
**Решение:** Ассеты хранятся в двух уровнях:  
```
data/assets/default/{category}/    ← базовые ассеты для всех систем
data/assets/systems/{name}/{category}/  ← ассеты конкретной системы
```
`GET /api/assets/{category}?system=shadowrun` возвращает мёрдженный список, где файл с одинаковым именем из `systems/` перекрывает `default/`.  
**Почему:** Иконка «fire.png» в D&D — классический огонь, в Shadowrun — неоновый кибер-эффект. Без override пришлось бы дублировать всю библиотеку для каждой системы.

### ADR-5: i18n Namespace Pattern
**Решение:** Два типа переводов с разными namespace:  
- `core` — UI строки панели (кнопки, заголовки, сообщения)  
- `systems/{name}` — термины конкретной НРИ-системы (hp/wounds/sanity, ac/defense/armor)  
Файлы лежат на бэке (`data/locales/{lang}/core.json`) и грузятся динамически через `i18next-http-backend`.  
**Почему:** Термины D&D и Shadowrun несовместимы. Namespace позволяет переключать только системные термины, не трогая UI.

### ADR-6: Encounter Save Format (полный стейт)
**Решение:** `POST /api/encounters/save` сохраняет:  
```json
{ "name": "...", "actors": [...], "history": [...], "round": N,
  "turn_queue": [...], "current_index": N, "is_active": true }
```
**Почему:** Сохранение только акторов (как в Roster) не даёт возобновить бой. Полный стейт позволяет продолжить с того же хода — актуально для многосессионных кампаний.  
**Следствие:** `POST /api/combat/load` восстанавливает всё включая очередь ходов и `is_active`.

### ADR-7: Background Thread для записи логов
**Решение:** `add_log()` пишет файлы логов в `threading.Thread(daemon=True)`, не блокируя HTTP-ответ.  
**Почему:** Запись на диск при каждом обновлении HP блокировала бы WebSocket broadcast. Daemon-thread гарантирует автоочистку при выходе.

### ADR-8: Filename Sanitization
**Решение:** `_safe_columns_filename()` и `_safe_system_dir()` чистят имена систем от опасных символов (`/\:*?"<>|&`, `..`), ограничивают длину до 100 символов.  
**Почему:** Имена систем вводит пользователь. «D&D 5e» должно сохраняться как `D_D 5e_columns.json`, а не падать с ошибкой ОС. Обратная совместимость сохранена через двойную проверку (raw + safe name).

## Отклонённые идеи

### ❌ SQLite вместо JSON-файлов
Google AI Studio оставил `better-sqlite3` в зависимостях. Оставлять не будем — для одного пользователя JSON-файлы проще, не требуют миграций, легко редактируются вручную и бэкапятся.

### ❌ Разделение на frontend и backend репозитории
Излишняя сложность для одного разработчика. Монорепо + `npm run dev`.

### ❌ Event Sourcing для Undo/Redo
Простые снэпшоты `history_stack[]` достаточны для 20 шагов. Event sourcing — over-engineering для данного масштаба.

### ❌ Хранение Undo-стэка в CombatState
Undo-стэк живёт только в памяти сервера и не нужен клиентам. Включение в модель раздуло бы WebSocket-сообщения.

## Ледник (Icebox) — фичи на будущее

### 🧊 Dice Engine
Модульная система бросков кубиков. Python-библиотека `d20` для D&D (парсинг `1d20+5`), пулы d6 для Shadowrun, кастомные пулы для WoD. UI-компонент «dice tray» прямо в трекере.

### 🧊 Google Sheets Export
Экспорт лога боя в Google Sheets для статистики кампании. Требует OAuth — сложно, откладываем.

### 🧊 Второй экран (Player View)
Отдельная страница `/player` — только публичные данные (revealed акторы, инициатива). Мастер может вывести на второй монитор или планшет игроков.

### 🧊 Community Localization
Папка `data/locales/` смонтирована как статика и задизайнена под community-вклад: пользователь кидает `data/locales/de/core.json` — и немецкий язык просто появляется в дропдауне.

### 🧊 Encounter Scripting
Скрипты на YAML/JSON для автоматизации событий: «на раунде 3 — заспавнить подкрепление», «при HP < 20% — сменить тактику».

### 🧊 Multiple Miniature Screens
Поддержка нескольких форматов экрана ESP32 (кроме 172×320). Список форматов в конфигурации кроппера.

## Тесты железа (ESP32)

- **Waveshare ESP32-C6-LCD-1.47** — целевой экран миниатюры (172×320px)
- Связь: Wi-Fi → WebSocket
- Рендер: `compositor.py` (Pillow) генерирует PNG под конкретный `MiniatureLayout`
- LED: планируется синхронизация с цветами фракций через `sync_led_to_ui`
- Питание: USB от павербанка или LiPo + зарядный модуль (решение в разработке)
