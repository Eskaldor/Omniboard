> Обновлено: 03.03.2026

## Принятые архитектурные решения

### ADR-1: Монорепо (frontend + backend в одном репо)
**Решение:** Оставить всё в одной папке. `src/` — фронт, `backend/` — питон.  

### ADR-2: React 19 вместо Vue 3
**Решение:** После перезапуска прототипа в Google AI Studio стек сменился с Vue 3 + Pinia на **React 19 + TypeScript**.  

### ADR-3: Dual History Pattern (два разных «history»)
**Решение:** В проекте существуют две независимые системы истории:  
1. `CombatState.history: List[LogEntry]` — нарративный лог.
2. `history_stack: list` + `history_index: int` в `main.py` — технические снэпшоты.

### ADR-4: Asset Override Pattern
**Решение:** Ассеты хранятся в двух уровнях (`default` и `systems/<name>`).

### ADR-5: i18n Namespace Pattern
**Решение:** UI-локализация (ядро) живет в `data/locales/{lang}/core.json`. Системная локализация живет внутри самих систем (см. ADR-10).

### ADR-6: Encounter Save Format (полный стейт)
**Решение:** `POST /api/encounters/save` сохраняет полный стейт (`actors`, `history`, `round`, `turn_queue`, `current_index`, `is_active`).

### ADR-7: Background Thread для записи логов
**Решение:** `add_log()` пишет файлы логов в `threading.Thread(daemon=True)`.

### ADR-8: Filename Sanitization
**Решение:** Очистка имен файлов через `_safe_columns_filename()` и `_safe_system_dir()`.

### ADR-9: AI Context Separation (Подготовка для Red Knight)
**Решение:** Механика, локализация и промпты для LLM строго разделены в разные файлы.

### ADR-10: System Encapsulation (Всё в одной папке)
**Решение:** Каждая игровая система полностью инкапсулирована в своей папке внутри `data/systems/<system_name>/` (Механика, Переводы, ИИ-профили). Базовые шаблоны являются Read-Only.

---

## Отклонённые идеи
- **SQLite вместо JSON-файлов:** Оставляем JSON — легко редактировать вручную.
- **Event Sourcing для Undo/Redo:** Over-engineering. Снэпшотов достаточно.

## Ледник (Icebox) — фичи на будущее

### 🧊 AI-Ассистент "Красный Рыцарь" (Red Knight)
Голосовой помощник для Мастера (локальная LLM + STT/TTS). Управление боем голосом, броски кубиков, автологирование. См. `Red_Knight_Vision.md`.

### 🧊 Комментарии и Макросы (Smart Comments)
1. **Текстовое поле (Comment Field):** Новый тип колонки `type: string/text`, который отображается в UI не как узкий input для цифр, а как широкое текстовое поле или tooltip. Идеально для заметок "У него зелье в кармане" или "Под баффом барда".
2. **Микро-скрипты в комментариях:** Возможность писать простейшие триггеры прямо в текстовых полях. Например: `[on_turn_start: reset_reaction]`, `[on_round_end: action_points += 2]`. Позволит автоматизировать рутину (восстановление реакций, очков действий) без хардкода в движок.

### 🧊 Бестиарий и Библиотека (Monster Library)
Автоматический парсинг баз данных (Open5e API, JSON дампы) и конвертация их в готовые Actor-шаблоны для быстрого Drag-and-Drop'а врагов в бой.

### 🧊 System Logic Engine (Движок правил)
Возможность хранить специфичную логику передачи хода или расчета инициативы в `logic.py` внутри папки системы.

### 🧊 Dice Engine
Модульная система бросков кубиков (`1d20+5`, пулы d6). UI-компонент «dice tray».

### 🧊 Player View (Второй экран)
Отдельная страница `/player` — только публичные данные.

### 🧊 Community RAG Database
Возможность экспорта и импорта готовых векторных баз данных правил для разных НРИ систем для обмена между мастерами.
