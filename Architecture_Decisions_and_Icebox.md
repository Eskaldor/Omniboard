> Обновлено: 03.03.2026

## Принятые архитектурные решения

### ADR-1: Монорепо (frontend + backend в одном репо)
**Решение:** Оставить всё в одной папке. `src/` — фронт, `backend/` — питон.  
**Почему:** Для одного разработчика монорепо быстрее в управлении. `npm run dev` запускает оба сервера через `concurrently`. Разделять смысла нет пока нет команды.  

### ADR-2: React 19 вместо Vue 3
**Решение:** После перезапуска прототипа в Google AI Studio стек сменился с Vue 3 + Pinia на **React 19 + TypeScript**.  

### ADR-3: Dual History Pattern (два разных «history»)
**Решение:** В проекте существуют две независимые системы истории:  
1. `CombatState.history: List[LogEntry]` — **нарративный лог** событий боя (что произошло)  
2. `history_stack: list` + `history_index: int` в `main.py` — **технические снэпшоты** для Undo/Redo (20 штук)  
**⚠️ Важно для разработки:** `history_stack` НЕ сериализуется в `CombatState`, он живёт только в памяти сервера.

### ADR-4: Asset Override Pattern
**Решение:** Ассеты хранятся в двух уровнях:  
`data/assets/default/{category}/` (база) и `data/assets/systems/{name}/{category}/` (перекрывают базу).

### ADR-5: i18n Namespace Pattern
**Решение:** UI-локализация (ядро) живет в `data/locales/{lang}/core.json`. Системная локализация живет внутри самих систем (см. ADR-10). Frontend грузит их через `i18next-http-backend`.

### ADR-6: Encounter Save Format (полный стейт)
**Решение:** `POST /api/encounters/save` сохраняет полный стейт (`actors`, `history`, `round`, `turn_queue`, `current_index`, `is_active`) для бесшовного возобновления боя.

### ADR-7: Background Thread для записи логов
**Решение:** `add_log()` пишет файлы логов в `threading.Thread(daemon=True)`.

### ADR-8: Filename Sanitization
**Решение:** Очистка имен файлов через `_safe_columns_filename()` и `_safe_system_dir()`.

### ADR-9: AI Context Separation (Подготовка для Red Knight)
**Решение:** Механика, локализация и промпты для LLM строго разделены в разные файлы, чтобы защитить код от лингвистического мусора и ускорить RAG/Function Calling.

### ADR-10: System Encapsulation (Всё в одной папке)
**Решение:** Каждая игровая система полностью инкапсулирована в своей папке внутри `data/systems/<system_name>/`. 
Внутри папки системы хранится всё:
- Механика (`columns.json`, `logic.py` в будущем).
- Локализация (`locales/<lang>.json`).
- ИИ-контекст (`ai_profiles/<lang>_schema.json`).
- Миничарники и макросы (в будущем).
Базовые шаблоны систем (например, `DnD5e`) являются **Read-Only**. Если пользователь пытается их изменить через UI (добавить стат), бэкенд автоматически форкает (копирует) систему с новым именем (например, `DnD5e_custom`).
**Почему:** Это делает систему портативной (для передачи комьюнити достаточно скинуть одну папку). Read-Only защищает базовые настройки от случайной поломки. Разделение файлов внутри папки сохраняет соблюдение ADR-9.

## Отклонённые идеи

### ❌ SQLite вместо JSON-файлов
Оставляем JSON — легко редактировать вручную и бэкапить.

### ❌ Event Sourcing для Undo/Redo
Over-engineering. Снэпшотов достаточно.

## Ледник (Icebox) — фичи на будущее

### 🧊 AI-Ассистент "Красный Рыцарь" (Red Knight)
Голосовой помощник для Мастера (локальная LLM + STT/TTS). Управление боем голосом, броски кубиков, автологирование. См. `Red_Knight_Vision.md`.

### 🧊 System Logic Engine (Движок правил)
Возможность хранить специфичную логику передачи хода или расчета инициативы в скриптах внутри папки системы.

### 🧊 Dice Engine
Модульная система бросков кубиков (`1d20+5`, пулы d6). UI-компонент «dice tray».

### 🧊 Player View (Второй экран)
Отдельная страница `/player` — только публичные данные.

### 🧊 Community Localization
Монтирование папки `data/locales/` как статики. Загрузка фанатских переводов `core.json`.

### 🧊 Encounter Scripting
Скрипты на YAML/JSON для автоматизации событий («на раунде 3 — заспавнить подкрепление»).

### 🧊 Multiple Miniature Screens
Поддержка ESP32 разных форматов.
