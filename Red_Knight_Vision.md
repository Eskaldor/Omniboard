# 🔴 Red Knight: AI-Powered GM Assistant (Vision)

## Концепция
Red Knight ("Красный Рыцарь" — в честь богини стратегии из Forgotten Realms) — это проект глубокой интеграции ИИ в аппаратно-программный комплекс Omniboard. 
Цель: создать голосового AI-ассистента ("боевую подругу Ингрис"), который снимает микроменеджмент с Мастера (GM), управляя боевым трекером с помощью голосовых команд, ведя интеллектуальное логирование и выступая как RAG-справочник по правилам НРИ.

Это **Moonshot Project**, который будет реализовываться после завершения основного функционала и стабильной работы Omniboard + ESP32. Однако архитектурный фундамент для него закладывается уже сейчас, на этапе i18n-локализации.

---

## 🛠 Технический Стек (Черновик)
- **STT (Распознавание голоса):** Локальный Whisper.cpp (base/small).
- **LLM (Мозг):** Локальная Llama 3 8B (или аналогичная легкая модель, ~4-8GB VRAM) для простых команд function calling + Облачный API (Claude/GPT) для сложных макросов и нарратива.
- **RAG (База знаний):** ChromaDB / FAISS для поиска по SRD и рулбукам (D&D 5e, Shadowrun и т.д.).
- **TTS (Озвучка):** Piper TTS (быстрый, локальный) или аналог с качественными русскими голосами.
- **Железо:** Опционально выделенный Mini PC (Raspberry Pi 5 / Intel NUC) с USB-микрофонами (петличка для GM, направленный для стола).

## 🏗 Архитектура: Мост между AI и Omniboard

Главная проблема: Разрыв между локализованным UI (то, что видит и говорит человек) и техническими ключами бэкенда (`hp`, `ac`).
*Нельзя заставлять LLM читать файлы UI-локализации.*

**Решение: AI System Profiles (Предсобранные контексты)**
Вместо динамической генерации схем или парсинга UI-текстов, для ИИ создаются статические "словари-инструкции" (System Prompts + Function Calling Tools) для каждой игровой системы.

### Структура данных (Тройное разделение):
1. **Механика (Без текста):** `data/systems/<system>/columns.json` — только структура (id, type).
2. **UI (Только текст):** `data/locales/<lang>/systems/<system>.json` — то, что рендерит React (name, short).
3. **AI Profile (Контекст для ИИ):** `data/ai_profiles/<lang>/<system>_schema.json` — статический файл-инструкция.

### Пример `data/ai_profiles/ru/dnd5e_schema.json`:
```json
{
  "system_context": "Ты Ингрис, помощник для системы D&D 5e. Термины 'броня', 'кд', 'класс защиты' всегда относятся к параметру 'ac'. Термины 'хиты', 'хп', 'здоровье' относятся к 'hp'.",
  "tools": [
    {
      "name": "modify_stat",
      "description": "Изменить параметр персонажа",
      "parameters": {
        "type": "object",
        "properties": {
          "stat_id": {
            "type": "string",
            "enum": ["hp", "ac", "speed"],
            "description": "hp (хиты, здоровье), ac (броня, кд), speed (скорость, перемещение)"
          }
        }
      }
    }
  ]
}
```

*Примечание для разработчиков: При рефакторинге или чистке кода **НЕ УДАЛЯТЬ** упоминания ai_profiles или структуры подготовки ИИ. Это заложенный фундамент для проекта Red Knight.*

## 🚀 Roadmap (Дорожная карта)
1. **Подготовка (Текущий этап):** Реализация чистой i18n-локализации в Omniboard, резервирование архитектуры под AI Profiles.
2. **Phase 1 (MVP):** STT wake word → парсинг команды локальной LLM → вызов API Omniboard → TTS подтверждение.
3. **Phase 2 (Rules RAG):** Загрузка SRD систем, вопросы по правилам во время сессии ("Ингрис, как работает Sneak Attack?").
4. **Phase 3 (Макросы):** Выполнение каскадных команд ("Ингрис, рыцари атакуют орка" -> авто-броски -> применение урона).
5. **Phase 4 (Авто-логирование):** Continuous listening, speaker diarization, автоматическая запись важных событий в лог на основе диалогов за столом.
